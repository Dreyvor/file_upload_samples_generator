# Security and Safety Guardrails

## Intended use

This project is intended for:

- authorized penetration tests;
- internal QA/security validation;
- developer education;
- regression testing of upload validation;
- evaluating upload pipelines and parser routing.

## Prohibited content

Do not implement generators for:

- PHP/ASP/JSP web shells;
- reverse shells;
- malware droppers;
- credential theft;
- destructive decompression bombs;
- uncontrolled denial-of-service payloads;
- exploit payloads copied from parser CVEs;
- files that phone home to third-party infrastructure.

## Allowed test content

Allowed:

- benign magic-byte samples;
- valid small files;
- invalid/truncated files;
- controlled malformed files;
- metadata markers;
- passive XSS reflection strings;
- local-only references;
- polyglot files made from benign seeds;
- bounded resource stress files.

## Risk levels

Use these risk levels in manifest:

```text
info       baseline and documentation samples
low        malformed, mismatch, metadata marker samples
medium     bounded stress, polyglot samples
disabled   potentially disruptive samples not generated by default
```

## Network behavior

Generated files must not make network requests by default.

If a future optional SSRF-marker category is added:

- disable by default;
- require `--enable-network-markers`;
- use only placeholder domains or tester-supplied URLs;
- mark entries as `medium`;
- document expected effect clearly.

## Resource behavior

Default generation limits:

```text
max_file_size_mb = 10
max_total_output_mb = 250
max_pixels = 25_000_000
max_tiff_pages = 5
mitra_timeout_seconds = 30
```

Do not generate compression bombs.

## Filenames

Risky filenames must appear only in:

```text
multipart-recipes/*.md
```

Do not create local files named:

```text
../../x.pdf
CON.jpg
NUL.pdf
file%0a.pdf
```

Instead, create safe local representative files and document raw multipart filename values.

## Third-party repositories

Do not vendor third-party payload files by default.

Prefer:

- generate benign local equivalents;
- optional import paths supplied by user;
- include provenance and license notes.

## Report-friendly output

Every manifest entry should explain:

- why the sample exists;
- what secure behavior looks like;
- what weak behavior looks like;
- what evidence the tester should collect.

Example secure behavior:

```text
Reject upload because extension .pdf does not match detected JPEG content.
```

Example weak behavior:

```text
Accept upload, label it as PDF, then image thumbnailer successfully processes it as JPEG.
```
